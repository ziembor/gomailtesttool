# gomailtesttool 3.0 Refactor — Implementation Plan

## Context
The repo currently ships **5 separate binaries** (`imaptool`, `jmaptool`, `msgraphtool`, `pop3tool`, `smtptool`), each with duplicated `main.go` (signal handling, logger setup) and duplicated `config.go` (stdlib `flag` parsing, env-var glue, `validateConfiguration()`). A second `src/go.mod` exists and holds a near-complete duplicate of `cmd/msgraphtool/` as dead code. Integration testing is PowerShell-only.

The original `migration/3.0_MIGRATION_PLAN.md` is directionally correct but (a) understates how much of `internal/common/` is already built, (b) overstates `src/`'s role (it's dead code, not shared utils), (c) proposes a `ProtocolClient` interface that doesn't fit — msgraph's action set (calendar events, availability, invite) has no IMAP/SMTP analogue, and (d) treats testcontainers migration as a one-phase drop-in when it's a standalone effort.

This plan replaces that one. It keeps the unified Cobra CLI goal but **drops the forced protocol interface**, **merges Phases 1+2**, and **scopes testing separately**.

## Guiding decisions

1. **No `ProtocolClient` interface.** Protocols share a *framework* (flag binding, logger init, signal handling, output formatting, context propagation), not a common operation surface. msgraph's 7 actions (`getevents`, `sendmail`, `sendinvite`, `getinbox`, `getschedule`, `exportinbox`, `searchandexport`) don't overlap with SMTP's 4 (`testconnect`, `teststarttls`, `testauth`, `sendmail`). Forcing an interface means the biggest method is `interface{}`.
2. **`src/` is deleted outright, not migrated.** It's a stale duplicate of `cmd/msgraphtool/` — not referenced by any live binary. Verify with grep, then delete the directory and `src/go.mod`.
3. **Cobra + Viper adopted, but per-protocol flag sets stay per-protocol.** Each subcommand registers its own flag set; only *global* flags (`--verbose`, `--log-file`, `--output`, `--config`) live at root.
4. **Backward compat: thin shim binaries for one release.** `cmd/smtptool/main.go` becomes a 10-line shim that `exec.Command`s `gomailtest smtp ...`. Users scripting old names keep working; shims removed in 3.1.
5. **Flag names preserved exactly** on first release to avoid churn. Rename only if Cobra parser rejects them.
6. **msgraph drives the design.** It's the outlier (Graph SDK, not net-style protocol) — build the Cobra skeleton against msgraph first; if it fits, the simpler protocols will fit trivially.
7. **Phase 5 (testing) is out of scope for this plan.** Create a stub document; execute separately.

## Work units

### Unit A — Kill the dead code (1 PR, ~1 day)
**Goal:** Remove `src/` and `src/go.mod`. Pure deletion.

Steps:
1. `grep -r "gomailtesttool/src" .` — verify zero live references outside `src/` itself. Current belief: the `cmd/*` binaries import only from `internal/`, and `src/` is standalone.
2. Delete `src/` directory and `src/go.mod` / `src/go.sum`.
3. Run `go build ./...` and `go test ./...` from root — must stay green.
4. Remove any top-level README section referencing `src/`.

**Ships independently to main.** No user-visible change.

### Unit B — Shared bootstrap package (1 PR, ~1 day)
**Goal:** Extract the identical boilerplate from all 5 `main.go` files into `internal/common/bootstrap/`.

Create `internal/common/bootstrap/bootstrap.go` with:
- `SetupSignalContext() (context.Context, context.CancelFunc)` — the identical signal handling block currently duplicated across `cmd/*/main.go`
- `InitLogger(toolName, action string, format logger.LogFormat, logFile string) (logger.Logger, error)` — wraps the current `logger.SetupLogger()` + `logger.NewLogger()` sequence

Each existing `cmd/*/main.go` then calls these instead of inlining. **No behavior change, no Cobra yet.**

Reuses:
- `internal/common/logger` (already slog-based via `internal/common/logger/slog.go`)
- `internal/common/logger/logger.go` `Logger` interface — unchanged

**Ships independently.** Pure refactor, no user-visible change. Proves the shared-code path works before Cobra lands on top.

### Unit C — Cobra skeleton + msgraph subcommand (1 PR, ~3 days)
**Goal:** Stand up `cmd/gomailtest/` with Cobra/Viper, port msgraph first as the hardest case.

Layout:
```
cmd/gomailtest/
  main.go                    # 10 lines: rootCmd.Execute()
  root.go                    # rootCmd + global flags (--verbose, --log-file, --output, --config)
internal/protocols/msgraph/
  cmd.go                     # NewCmd() returns *cobra.Command with all msgraph subcommands
  config.go                  # MOVED from cmd/msgraphtool/config.go; Viper-bound
  auth.go                    # MOVED from cmd/msgraphtool/auth.go (unchanged)
  handlers.go                # MOVED from cmd/msgraphtool/handlers.go (unchanged)
  handler_mail.go            # MOVED (unchanged)
  handler_calendar.go        # MOVED (unchanged)
  handler_search.go          # MOVED (unchanged)
```

Key design points:
- Each msgraph action becomes a **Cobra subcommand leaf**: `gomailtest msgraph sendmail`, `gomailtest msgraph getevents`, etc. Not a single `msgraph --action=X`. This replaces the current `executeAction()` dispatch switch with Cobra's native routing.
- Flags bound via Viper with three-level precedence: flag > env > config file. Env var names preserved exactly (`MSGRAPHTENANTID`, etc.) via explicit Viper bindings.
- `cmd/msgraphtool/main.go` becomes a 10-line shim: `os.Args = rewriteArgs(os.Args); gomailtest.Execute()`. Old users unaffected.
- Root-level `PersistentPreRunE` calls `bootstrap.SetupSignalContext()` + `bootstrap.InitLogger()`, stashes them in a context, child commands pull from context.

Deliberately NOT in this unit:
- No `ProtocolClient` interface
- No other protocols ported
- No new tests (existing 30 unit tests must still pass)

**Validation:** all current `msgraphtool --action=sendmail --tenant-id=...` invocations must produce identical behavior via either `msgraphtool ...` (shim) or `gomailtest msgraph sendmail ...` (new).

### Unit D — Port remaining 4 protocols (1 PR each, ~1 day each)
**Goal:** Repeat Unit C's pattern for `smtp`, `imap`, `pop3`, `jmap`.

These are mechanically simpler than msgraph. Each PR:
1. Creates `internal/protocols/<proto>/cmd.go` exposing `NewCmd()`
2. Moves `cmd/<proto>tool/config.go` → `internal/protocols/<proto>/config.go`, Viper-bound
3. Converts `executeAction()` switch into Cobra subcommand leaves
4. Replaces `cmd/<proto>tool/main.go` with shim
5. Registers `NewCmd()` in `cmd/gomailtest/root.go`

Order: **smtp → pop3 → imap → jmap** (simplest action sets first).

### Unit E — Consolidate READMEs (1 PR, ~half day)
**Goal:** Replace 6 tool-specific READMEs with one root README + `docs/protocols/<proto>.md` per protocol.

- `README.md` gets a "Commands" section linking to per-protocol docs
- `IMAPTOOL_README.md`, `JMAPTOOL_README.md`, `MSGRAPHTOOL_README.md`, `POP3TOOL_README.md`, `SMTP_TOOL_README.md` → `docs/protocols/{imap,jmap,msgraph,pop3,smtp}.md`
- All command examples updated from `smtptool --flag` to `gomailtest smtp <action> --flag`, with a "legacy binary names" note pointing at the shims
- `BUILD.md`, `EXAMPLES.md`, `TROUBLESHOOTING.md`, `SECURITY.md`, `UNIT_TESTS.md`, `INTEGRATION_TESTS.md` get search-and-replace for command examples

### Unit F — Remove shims (3.1 release, not this plan)
After one release cycle, delete `cmd/{imap,jmap,msgraph,pop3,smtp}tool/` directories entirely. Only `cmd/gomailtest/` remains.

### Deferred — Testing modernization
Create `migration/3.0_TESTING_PLAN.md` as a stub with: "testcontainers-go migration to be designed separately; tracked as own effort." Existing 30 unit tests + PowerShell integration scripts stay in place through 3.0 release.

## What explicitly does NOT happen in 3.0

- No `ProtocolClient` / `Authenticator` / `MailLister` interfaces (over-abstraction for the action surface)
- No testcontainers-go (own effort, deferred)
- No flag renames (preserve exact names)
- No deletion of legacy `cmd/*tool/` directories (kept as shims until 3.1)
- No new output formats (`text`/`json`/`csv` already supported via `OutputFormat`/`LogFormat`)
- No Phase-6 global refactor — each unit ships to main independently

## Critical files

**Will move (msgraph, Unit C):**
- `cmd/msgraphtool/{main,config,auth,handlers,handler_mail,handler_calendar,handler_search}.go` → `internal/protocols/msgraph/` (main.go becomes shim)

**Will move (other protocols, Unit D):**
- `cmd/{smtp,imap,pop3,jmap}tool/*.go` → `internal/protocols/{smtp,imap,pop3,jmap}/`

**Will be created:**
- `cmd/gomailtest/main.go`, `cmd/gomailtest/root.go`
- `internal/common/bootstrap/bootstrap.go` (Unit B)
- `internal/protocols/<proto>/cmd.go` × 5

**Will be deleted:**
- `src/` directory and `src/go.mod` / `src/go.sum` (Unit A)

**Reused as-is (no changes):**
- `internal/common/logger/` — already slog-based; `Logger` interface stays
- `internal/common/{retry,ratelimit,validation,security,version}` — already shared
- `internal/{imap,jmap,pop3,smtp}/protocol/` — protocol implementations unchanged

## Verification per unit

**Unit A:** `go build ./...` + `go test ./...` green after `src/` deletion.

**Unit B:** All 5 existing binaries build and pass their current tests; diff their `main.go` files — each should shrink by ~30 lines.

**Unit C:**
- `go build ./cmd/gomailtest && go build ./cmd/msgraphtool`
- For each of the 7 msgraph actions, run the command two ways and confirm identical behavior:
  - Old: `./msgraphtool --action=sendmail --tenant-id=... --client-id=... --from=... --to=... --subject=test`
  - New: `./gomailtest msgraph sendmail --tenant-id=... --client-id=... --from=... --to=... --subject=test`
- Env var precedence: set `MSGRAPHTENANTID=xxx`, omit `--tenant-id`, confirm pickup via Viper binding
- Signal handling: send SIGINT mid-run, confirm graceful shutdown (same as before)
- Existing `cmd/msgraphtool/*_test.go` tests pass against moved code (update import paths only)

**Unit D:** Same verification recipe per protocol — each protocol's existing unit tests must pass after the move; each action's old and new invocation must produce identical output.

**Unit E:** `grep -r "smtptool \|imaptool \|pop3tool \|jmaptool \|msgraphtool " README.md docs/ BUILD.md EXAMPLES.md TROUBLESHOOTING.md` → zero hits (all examples updated).

## Open questions to resolve before Unit C

1. **Shim strategy confirmation.** Thin shims via `exec.Command`, or direct in-process `rootCmd.SetArgs()` dispatch? In-process is cleaner but requires `gomailtest` binary to be discoverable. Recommend in-process.
2. **Config file format.** Viper supports YAML/JSON/TOML — pick one for `--config`. Recommend YAML (standard for Go CLIs).
3. **Env var prefix.** Current: per-tool (`MSGRAPHTENANTID`, `SMTPHOST`). Viper default would prefer `GOMAILTEST_MSGRAPH_TENANT_ID`. Preserve old names via explicit bindings, or introduce new ones with deprecation warning? Recommend preserve.

These are design details, not blockers — surface them at Unit C kickoff.
