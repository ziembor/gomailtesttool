# gomailtesttool 3.0 Migration Plan

This document outlines the architectural changes and migration steps for the 3.0 release of `gomailtesttool`. The primary goals are to reduce code duplication, standardize the project structure, and provide a unified CLI experience.

## Goals
- **Unified CLI**: A single binary (`gomailtest`) with subcommands for each protocol.
- **Reduced Duplication**: Shared logic for logging, configuration, and signal handling.
- **Standard Layout**: Adherence to standard Go project structures (`cmd/`, `internal/`).
- **Improved Testability**: Protocol interfaces and Go-native integration tests.

---

## Phase 1: Project Restructuring & Module Consolidation
*Target: Simplify dependency management and align with Go standards.*

1. **Consolidate Modules**:
   - Merge `src/go.mod` dependencies into the root `go.mod`.
   - Delete `src/go.mod` and `src/go.sum`.
   - Update all internal import paths from `gomailtesttool/src/...` to `gomailtesttool/internal/...` (or appropriate new paths).
2. **Reorganize Directories**:
   - Create `cmd/gomailtest/` for the main entry point.
   - Move protocol-specific logic from `cmd/[protocol]tool/` to `internal/protocols/[protocol]/`.
   - Move `src/msgraphtool.go` and related MSGraph files to `internal/protocols/msgraph/`.
   - Move shared utilities (`logger.go`, `utils.go`, etc.) from `src/` to `internal/common/`.

---

## Phase 2: Unified CLI Framework (Cobra)
*Target: Replace multiple binaries with a single, subcommand-based CLI.*

1. **Initialize Cobra**:
   - Add `github.com/spf13/cobra` and `github.com/spf13/viper`.
2. **Define Command Hierarchy**:
   - `gomailtest` (Root)
     - `imap` (Subcommand)
     - `jmap` (Subcommand)
     - `msgraph` (Subcommand)
     - `pop3` (Subcommand)
     - `smtp` (Subcommand)
3. **Global Flags**:
   - `--verbose`, `-v`: Global verbosity level.
   - `--config`: Path to a configuration file (YAML/JSON).
   - `--log-file`: Path to write logs.
   - `--output`: Output format (`text`, `json`, `csv`).

---

## Phase 3: Abstraction & Shared Logic
*Target: Eliminate boilerplate and ensure consistent behavior.*

1. **Shared Logger**:
   - Refactor `internal/common/logger` to use `log/slog`.
   - Ensure the root command initializes the logger based on global flags.
2. **Centralized Configuration**:
   - Implement `internal/config` to handle flag parsing and environment variable binding.
3. **Protocol Interfaces**:
   - Define common interfaces in `internal/common/interfaces.go` (e.g., `ProtocolClient`, `Authenticator`, `MailLister`).

---

## Phase 4: Protocol Migration
*Target: Port existing protocols to the new framework.*

1. **Refactor Handlers**:
   - Update protocol-specific handlers to accept shared configuration and logger instances.
2. **Standardize Output**:
   - Implement a unified output formatter for consistent console and machine-readable output.
3. **Graceful Shutdown**:
   - Implement unified signal handling (SIGINT, SIGTERM) at the root level.

---

## Phase 5: Modernize Testing
*Target: Transition to Go-native integration tests.*

1. **Go Integration Tests**:
   - Replace PowerShell scripts with Go tests using `testcontainers-go`.
   - Use containers for local mail servers (e.g., MailHog for SMTP, GreenMail for IMAP/POP3).
2. **Mocking**:
   - Leverage new protocol interfaces to create comprehensive unit tests with mocks.

---

## Phase 6: Documentation & Final Cleanup
*Target: Update documentation and remove legacy code.*

1. **Consolidate READMEs**:
   - Create a single `README.md` at the root.
   - Move protocol-specific details to `docs/protocols/`.
2. **Update Examples**:
   - Provide updated command examples (e.g., `gomailtest smtp send ...`).
3. **Remove Legacy Files**:
   - Delete old `cmd/[protocol]tool/` directories and unused files in `src/`.
